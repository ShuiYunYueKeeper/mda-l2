## 背景与目标

### 背景

需要实现一个 Markdown 批注管理工具（MDA），通过 Markdown 标准注释语法（`[comment]: <>`）在 `.md` 文件中嵌入结构化批注数据，支持批注的增删改查。批注以 `@anno` 为前缀、JSON 为数据载体，渲染后完全不可见，不影响文档正常阅读。工具需同时提供 GUI 客户端（`mda`）和 CLI 命令行（`mda-cli`）两种使用方式。

### 目标

1. 完整实现 CLI 4 个命令：`scan`（扫描/筛选）、`add`（添加）、`edit`（编辑）、`remove`（删除）
2. 完整实现 GUI 4 大模块：菜单栏、Markdown 渲染预览区（含批注色条 + 点击互定位）、批注面板（列表 + 筛选 + 增删改）
3. 批注增删改后正确写回 `.md` 文件，正文内容不受影响，渲染效果前后一致
4. 交付完整仓库，满足 L2 认证的 AI 协作资产沉淀、任务规划、交付质量保障要求

### 成功指标（可核验）

| 维度 | 指标 |
|------|------|
| CLI 功能完整性 | `scan`/`add`/`edit`/`remove` 四个命令均可正常执行，`--format json` 输出严格符合规范 |
| CLI 输出规范 | stdout 仅 JSON，stderr 走日志；无批注时输出 `[]` |
| GUI 功能完整性 | 打开文件→渲染预览→查看批注→筛选→增删改→重新加载，全流程可用 |
| 批注语法一致性 | 所有读写的批注严格符合 `[comment]: <> (@anno {...})` 格式 |
| 源文件保护 | 批注操作前后的非批注行逐字节一致（diff 无差异） |
| 可运行性 | 按 README ≤3 步完成环境准备并启动 GUI 和 CLI |

## 功能描述（详细）

### 批注数据模型

| 字段 | 类型 | 必填 | 约束 |
|------|------|------|------|
| `id` | string (UUID v4) | ✓ | 全局唯一，编辑/删除的精确定位键 |
| `content` | string | ✓ | 批注正文，多行用 JSON `\n` 转义 |
| `tags` | string[] | ✓ | 标签列表，如 `["bug","ui"]` |
| `level` | enum | ✓ | `critical` / `major` / `minor` / `info` |
| `status` | enum | ✓ | `open` / `resolved` / `wontfix` |
| `created_at` | string (ISO 8601) | ✓ | 带时区偏移，如 `2026-06-05T09:00:00+08:00` |

### 批注存储格式

```
[comment]: <> (@anno {"id":"...","content":"...","tags":[...],"level":"...","status":"...","created_at":"..."})
```

- 单行，放在被批注段落**正上方**
- 段落可有多条批注，依次排列在段落上方
- `@anno` 前缀区别于普通注释
- 渲染后完全不可见（标准 Markdown 注释语法）

### CLI 模式（mda-cli）

| 命令 | 签名 | 行为 |
|------|------|------|
| `scan` | `mda-cli scan <file\|dir> [-r] [--format json] [--status <s>] [--level <l>]` | 扫描文件/目录中所有批注；默认表格输出（ID/文件/行号/段落摘要/批注摘要/级别/状态）；`--format json` 输出 JSON 数组；`-r` 递归目录 |
| `add` | `mda-cli add <file> <line> <content> [--tags <t>] [--level <l>]` | 在 `<line>` 段落上方插入批注行；自动生成 UUID 和时间戳；`--level` 默认 `info`；`--tags` 逗号分隔 |
| `edit` | `mda-cli edit <file> <id> [--content <c>] [--tags <t>] [--level <l>] [--status <s>]` | 按 ID 查找批注并原地更新指定字段 |
| `remove` | `mda-cli remove <file> <id>` | 按 ID 查找并删除该批注行 |

**JSON 输出规范（`--format json`）**：

- 顶层 `[...]` 数组，无批注时 `[]`
- 每个元素含 8 字段：`id`, `file`, `line`, `content`, `tags`, `level`, `status`, `created_at`
- stdout 仅 JSON，日志/警告/错误走 stderr
- `file` 为字符串路径，`line` 为 1-based 行号

### GUI 模式（mda）

**启动方式**：`mda`（空窗口，菜单打开文件）；`mda <file>`（直接打开）

**窗口标题**：`MDA - 文件名.后缀`

| 模块 | 子项 | 详细规格 |
|------|------|----------|
| 菜单栏 | 文件→打开 | 系统文件选择器，筛选 `.md` |
| | 文件→重新加载 | 重新读取当前文件，同步外部修改 |
| 预览区（60-70%宽） | Markdown 渲染 | **基准：CommonMark 0.31 + GFM 表格扩展**。必须支持：H1-H6、段落/换行、粗体、斜体、行内代码、无序列表(`-/*/+`)、有序列表(`1.`)、链接(Ctrl+点击可打开外部浏览器)、图片(找不到时显示 alt 文本占位符)、引用块(`>`)、围栏代码块(区分样式)、水平线(`---`)、表格(管道语法) |
| | 批注色条 | 有批注段落左侧色条：红=critical、橙=major、黄=minor、灰=info |
| | 交互 | 点击段落→批注面板定位到对应批注；点击批注→预览区滚动到对应段落 |
| 批注面板（30-40%宽） | 列表 | 每条显示：内容、标签(彩色徽章)、级别、状态、时间 |
| | 筛选器 | 按状态、按级别、按标签 |
| | 操作 | 编辑按钮→弹窗修改字段；删除按钮→确认后删除；添加按钮→行号默认当前光标位置 |

### 不支持

- Markdown 编辑器（预览只读）
- 数学公式、Mermaid、脚注、任务列表复选框、定义列表

## 验收标准（Given/When/Then）

- **AC-1**：Given 一个包含批注的 `.md` 文件，When 执行 `mda-cli scan <file> --format json`，Then stdout 输出合法 JSON 数组，每个元素含 8 个必需字段，stderr 无 JSON 混入。
- **AC-2**：Given 一个不含批注的 `.md` 文件，When 执行 `mda-cli scan <file> --format json`，Then stdout 输出 `[]`，exit code = 0。
- **AC-3**：Given 一个 `.md` 文件，When 执行 `mda-cli add <file> <line> "<content>" --tags bug --level major`，Then 文件在指定段落上方新增一行 `[comment]: <> (@anno {...})`，其中 id 为 UUID、created_at 为当前时间、level=major、tags=["bug"]，正文行没有任何变化。
- **AC-4**：Given 一个包含批注的 `.md` 文件，When 执行 `mda-cli edit <file> <id> --status resolved`，Then 该批注的 status 字段更新为 `resolved`，其余字段不变，其余行不变。
- **AC-5**：Given 一个包含批注的 `.md` 文件，When 执行 `mda-cli remove <file> <id>`，Then 该批注行被删除，其余行不变。
- **AC-6**：Given 启动 GUI `mda`（无参数），When 窗口打开，Then 显示空预览区和空批注面板，窗口标题为 `MDA`，菜单栏可见。
- **AC-7**：Given GUI 已打开一个含批注的 `.md` 文件，When 查看预览区，Then Markdown 按 CommonMark 0.31 + GFM 表格规范正确渲染（标题/列表/粗斜体/代码块/表格/引用/链接/图片），有批注段落左侧显示对应级别的色条（红/橙/黄/灰）。
- **AC-7a**：Given GUI 已打开一个含批注的 `.md` 文件，When 检查渲染后的 HTML 内容，Then 批注数据（`@anno` JSON）不出现在任何渲染输出中——即渲染结果与删除所有 `[comment]: <> (@anno ...)` 行后的渲染结果逐像素一致。
- **AC-7b**：Given GUI 预览区渲染了一个含图片语法 `![alt](src)` 的文档，When 图片无法加载（路径无效/不存在），Then 显示 alt 文本作为占位符，不显示空白或损坏图标。
- **AC-7c**：Given GUI 预览区渲染了一个含链接 `[text](url)` 的文档，When Ctrl+点击该链接，Then 在系统默认浏览器中打开该 URL。
- **AC-8**：Given GUI 已打开文件，When 在批注面板使用筛选器选择 `status=open` 和 `level=critical`，Then 列表仅显示同时满足两个条件的批注。
- **AC-9**：Given GUI 已打开文件，When 点击预览区某个带批注的段落，Then 批注面板滚动并高亮对应的批注项。
- **AC-10**：Given GUI 已打开文件，When 点击批注面板中某条批注，Then 预览区滚动到对应段落并高亮。
- **AC-11**：Given GUI 已打开文件，When 通过批注面板编辑某条批注并保存，Then 文件写回更新后的批注 JSON，重新加载后修改生效。
- **AC-12**：Given GUI 已打开文件，When 通过批注面板删除某条批注并确认，Then 文件写回删除该批注行，重新加载后批注消失。
- **AC-13**：Given 任意批注操作（CLI add/edit/remove 或 GUI 编辑/删除/添加）完成后，When 对比操作前后文件的非批注行，Then 逐字节完全一致（仅批注行有变化）。
- **AC-14**：Given 不存在的文件路径，When 执行 CLI 命令或 GUI 打开，Then CLI stderr 报错 + exit code ≠ 0；GUI 弹窗提示文件不存在。

## 接口契约

### 不变更

| 条目 | 说明 |
|------|------|
| 批注语法格式 | 严格 `[comment]: <> (@anno {JSON})`，不可用 HTML comment `<!-- -->` 或其他变体 |
| JSON 字段名 | `id/content/tags/level/status/created_at` 字段名不可改变 |
| 主程序命名 | GUI 入口必须含 `mda`，CLI 入口必须含 `mda-cli` |
| 入口行为 | `mda` 无参→空窗口；`mda <file>`→直接打开 |

### 允许变更

| 条目 | 说明 |
|------|------|
| 技术栈选择 | 语言/框架/GUI 库不限，满足跨平台和可运行性即可 |
| 入口形式 | Python 脚本/Node bin/编译产物均可，README 中说明启动方式 |
| 内部模块划分 | 解析器/存储层/GUI 组件的内部架构自由设计 |
| 测试框架 | 任意标准测试框架，能统计覆盖率即可 |
| 非批注注释 | 文件中已有的普通 `[comment]: <>`（无 `@anno`）保留不动 |

## 风险与不确定性

| 风险 | 影响 | 缓解 |
|------|------|------|
| 批注解析正则遗漏边界 case（空行间、文件尾、特殊字符） | 中—批注漏识别或错归属 | 覆盖 20 个边界场景的单元测试（E1-E20），CI 中强制执行 |
| GUI 框架选择不当导致跨平台问题或启动复杂 | 高—不满足 NF-1/NF-2 | 优先选成熟跨平台方案（Electron/Tauri/PyWebView），预验证 Windows 启动流程 |
| Markdown 渲染库不支持全部 11 种必需语法 | 低—可能需要补充自定义渲染 | 预选库后以 CommonMark 0.31 官方 spec 用例逐项验证；缺失项自实现 |
| 文件写回时意外修改非批注行（换行符、编码等） | 高—违反强制约束 C2 | 每次写回后 diff 验证，单元测试覆盖 |
| 大型文件解析性能不足导致 GUI 卡顿 | 低—影响用户体验 | 使用流式/行级解析，避免全量 AST |
| L2 认证对 AI 协作记录的要求理解偏差 | 中—交付被打回 | 保留完整 prompt 记录，≥3 轮对话存为 docs/prompts/*.md |

### 对抗自检（反向追问）

1. 这个需求中有哪些地方是「推测」的而非原文明确说明的？ — GUI 中编辑/删除后的写回时机（实时写回 vs 缓存后批量保存）原文未明确，假设为实时写回（A6）。`line` 参数在段落有多行时的具体指代（任意行 or 首行）也是推测（A4）。
2. 如果产品验收时说"这不是我要的"，最可能是因为哪一点理解有误？ — 批注与段落的归属关系：如果段落定义（空行分隔 vs 非空行块）与验收方理解不一致，批注归属会出错。这需要在实现前用示例文件明确约定。
3. 有没有遗漏的用户场景或边界条件？ — (a) 批注 content 中包含 `@anno` 字符串本身是否会被误解析？(b) 两个段落紧邻（无空行）时各自批注如何区分？(c) 文件编码非 UTF-8 时的处理？(d) Windows 下 CRLF vs LF 换行符处理？
