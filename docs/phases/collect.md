> **路径约定**：下文 `{task_dir}/` 指 conductor 派发时声明的任务目录（如 `workspace/tasks/feature-YYMMDD-xxx/`）。

## 角色约束

- agents/pm.md（prompt 内已引用）
- ⚠️ 派发时须设置 `readonly: false`（需写入 thinking/P1-requirement.md）
- 推荐 subagent_type：`generalPurpose`（需要灵活搜索+写文件+提问）

## 输入文档

- 用户的需求描述（由 conductor 在 prompt 中传入）

## P1: 需求解读 Prompt

```
【必须先执行】读取以下文件并遵循其中的约束：
- agents/pm.md

你的任务分三步：先调研，再提问，最后写文档。不允许跳过调研直接写 thinking/P1-requirement.md。

## Step 0: 文档完整性确认（不可跳过）

在开始需求调研前，确认以下条件：
- [ ] 需求文档内容已完整获取（非截断、非部分截图）
- [ ] 如有附件（页面原型、埋点字典等），已获取或标注"待用户提供"

如任一不满足 → 返回给 conductor，请求补充文档。不得在文档不完整时继续 Step 1。

## Step 1.1: 需求调研（不可跳过）

在写任何需求文档之前，必须先完成以下调研：

1. **知识库检索**：读取 `skills/knowledge-loader/SKILL.md` 并按指引执行 `index` 获取模块索引，然后按索引执行 `module {module}` 加载相关模块的知识文档，了解现有架构
2. **历史任务搜索**：搜索 workspace/tasks/ 中是否有类似需求/功能的历史实现
3. **代码快速扫描（仅限背景确认）**：
   - 搜索起点：优先从 `{repo_root}/` 下搜索（conductor 会在 Task prompt 中提供 repo_root）。如仓库含多个子目录（如 Coding/、src/），先用 `ls` 或 `Glob` 确认顶层目录结构，再进入相关子目录搜索
   - ✅ 允许：确认目标模块是否存在、是否有同名/类似功能的历史实现、模块大致职责
   - ❌ 禁止：分析代码逻辑细节、读取具体实现（超时值、信号槽连接等）、验证技术假设
   - **活跃度鉴别**（搜索到多个同名/相似模块时必做）：对每个候选模块执行 `git log --oneline --since="6 months ago" -- <模块路径>`（注意：若目标在子仓库中，需先 cd 到子仓库执行）。近 6 个月无提交的模块标记为"疑似废弃/低活跃"，优先选择活跃模块。将鉴别结果写入 P1-requirement-research.md
   - 深度代码分析属于 P2 阶段职责
4. **竞品/行业参考**：如需求涉及常见功能（如隐私保护、权限管理），简要列出行业通用做法

将调研结果写入 {task_dir}/thinking/P1-requirement-research.md，作为后续需求文档的依据。

## Step 1.2: 提问清单（遇到不确定必须提问）

基于调研结果，列出所有不确定的点，**必须先向用户提问，等回答后再写需求文档**。

提问原则：
- 不确定 → 提问（而非猜测）
- 模糊 → 提问（而非自行解读）
- 有多种理解 → 列出选项请用户选择

提问格式：
  ```
  ## 需要澄清的问题
  1. [问题] — 我的初步理解是 X，但也可能是 Y。需要确认。
  2. [问题] — 需求文档没有提及 Z 场景，是否需要覆盖？
  ```

⚠️ 如果不确定的点超过 3 个，**必须先暂停输出 thinking/P1-requirement.md**，将问题列表呈现给用户。

## Step 1.3: 需求文档编写

在调研完成且关键问题已获回答后，才能写 thinking/P1-requirement.md。

思考方式（#1 CoT）：先列出你的分析思路（这个需求涉及哪些用户场景？核心目标是什么？有哪些隐含约束？），再逐步给出结论，不允许跳步直接输出最终结果。

输入：{用户的需求描述} + {task_dir}/thinking/P1-requirement-research.md + 用户回答

产出：读取模板 conductors/dev/templates/requirement.template.md，按其结构填充，写入 {task_dir}/thinking/P1-requirement.md（不允许 TODO 占位符）
文件末尾附：
- 置信度（百分比）+ 不确定的部分 + 是否需要人工澄清
- 确认状态标记：
  ```
  ## 确认状态
  状态: 待确认
  ```

## 关键假设清单（必填）

在 thinking/P1-requirement.md 中必须包含以下章节：

  ```
  ## 关键假设
  | # | 假设内容 | 证据来源 | 验证状态 |
  |---|---------|---------|---------|
  | A1 | 例：用户投屏时会切换到文件列表 | 需求文档原文 | ✅ 已确认 |
  | A2 | 例：投屏只包含硬件投影场景 | 无明确说明 | ❓ 待澄清 |
  ```

对每个❓假设，必须提出具体的澄清问题。

## 对抗自检（反向追问）

完成需求文档后，必须回答以下问题：
1. 这个需求中有哪些地方是我「推测」的而非原文明确说明的？
2. 如果产品验收时说"这不是我要的"，最可能是因为哪一点理解有误？
3. 有没有遗漏的用户场景或边界条件？

将上述追问的结论追加到 thinking/P1-requirement.md 的「风险与不确定性」章节。

质量门禁（自检后才能继续）：
- [ ] Step 1.1 调研已完成，thinking/P1-requirement-research.md 已写入
- [ ] Step 1.2 中不确定的点已全部向用户提问并获得回答（或标注"用户确认跳过"）
- [ ] 需求描述完整（背景、目标、用户故事）
- [ ] 每条验收标准都是 Given/When/Then 格式
- [ ] 无 TODO 占位符
- [ ] 已标注置信度和不确定部分
- [ ] 关键假设清单已列出，每条标注验证状态
- [ ] 对抗自检 3 个问题已回答

⛔ P1 到此结束。将 thinking/P1-requirement.md 呈现给用户等待确认，用户确认后才能进入后续阶段。
```
