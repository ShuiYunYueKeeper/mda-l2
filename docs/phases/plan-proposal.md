# P4-Step1: 并行方案生成（Best-of-N 独立提案）

> 本文件由 conductor 在 P4-Step1 时读取（dev 方案确定的第一步）。
> 本阶段并行派发 **K** 个 subagent（默认 **K=3**，范围 **3～5**），每个 subagent 收到**完全相同的 prompt**（与本文件内「P4-Step1 Prompt」代码块一致）。
> **路径约定**：下文 `{task_dir}/` 指 conductor 派发时声明的任务目录（如 `workspace/tasks/feature-YYMMDD-xxx/`）。
> 公共约定详见 `common/common-conventions.md`。

## 角色约束

- agents/cpp-architect.md（prompt 内已引用）
- ⚠️ 派发时须设置 `readonly: false`（需写入 `thinking/P4-proposal-{i}.md`）
- 推荐 subagent_type：`cpp-architect`

## 输入文档

- `{task_dir}/thinking/P2-design.md` — 已确认的设计方案
- `{task_dir}/thinking/P1-requirement.md` — 需求文档
- `{task_dir}/thinking/P3-enhanced-context.md` — 如存在则必须读取

## 并行策略：统一视角（Best-of-N）

所有 **K** 个 subagent 收到**相同的** phase 正文与相同的必读输入列表，各自**独立**产出一份实施计划草案。

**目的**不是做多视角差异化探索，而是用 **Best-of-N** 做一致性校验：多个独立 agent 若结论趋同，可提高对该实施取向的信心；若在接口切分、依赖顺序、风险判断等处**系统性分歧**，则在 Step2（蒸馏）中标记为需重点关注的张力或不确定性信号。

**独立性约束（强制）**：单个 subagent **不得**读取、引用或假设任何其他并行产出的 `P4-proposal-*.md`（含编号相邻文件）；不得依据「多数派」调整结论。仅允许依据 P1/P2/P3 与仓库内静态证据推理。

Conductor 在派发 Task 时注入：**提案序号 `i`**（连续整数 **1…K**）及目标路径 `{task_dir}/thinking/P4-proposal-{i}.md`。除序号与路径外，不得向不同 Task 注入不同的视角标签或差异化指令。

## P4-Step1 Prompt

```
【必须先执行】读取以下文件并遵循其中的约束：
- agents/cpp-architect.md
- rules/common/wps-cpp-spec/wps-cpp-spec.mdc
- rules/common/wps-cmake-spec/wps-cmake-spec.mdc（涉及 CMakeLists.txt 时）

你是 P4-Step1 并行提案链路中的第 **{i}** 号独立提案者（Best-of-N）。你将与其他并行 agent 收到相同指令；你的产出仅写入指定文件，且**严禁**读取或引用任何 `P4-proposal-*.md`（含其他编号）。

必读输入（按顺序完整阅读）：
- {task_dir}/thinking/P2-design.md
- {task_dir}/thinking/P1-requirement.md
- 若存在：{task_dir}/thinking/P3-enhanced-context.md

⛔ **独立性**：不得假设其他提案的内容；不得在文中写「与其他提案一致/不一致」之类表述（你不知道其他提案）；仅基于上述输入与允许的静态检索撰写。

### 允许的静态工具

Read / Grep / SemanticSearch、知识库与 phase 授权的工程扫描类 Skill（若有）；禁止运行时探测、动态 MCP 探测类验证替代设计推断（与本阶段职责无关时可不写 POC）。

引用宜**精简**：涉及设计方案处用章节锚点或小节标题指向 `P2-design.md`，避免全文复述。

---

## 产出文件头部元信息（写入正文最前）

在 `{task_dir}/thinking/P4-proposal-{i}.md` 第一行起依次给出：

1. 一级标题：`# P4 实施计划草案 — 提案 {i}`
2. 短段落声明：**提案序号**、`Best-of-N 统一视角`、**独立完成**（未读取其他 proposal）
3. **与设计对齐摘要**：用 5-10 条要点说明本草案如何落实 `P2-design.md` 的推荐方向；若有你认为必要的偏离，逐条说明理由与代价。

---

## 第一章：接口定义（接口先行）

本节必须存在。涉及新增类、公共 API、跨模块契约时给出可实施的声明级定义；若无新增公共接口，明确写「无新增公共接口」并简述依据。

若涉及新增接口，至少包含：头文件级签名要点、错误契约（错误码或异常语义）、必要的兼容性说明（前向/后向）、若适用则包含信号槽或异步边界。

---

## 第 1.5 章：方案架构图（Mermaid）

用 Mermaid 图展示本提案的整体架构设计。要求：
- 展示涉及的模块/组件及其交互关系
- 标注通信机制（直接调用 / IPC / signal-slot / REST / event）
- 高亮新增/修改的节点（用 `:::new` 或备注标注）
- 如涉及跨进程/跨服务通信，用不同 participant 区分进程边界
- 如仅涉及单模块内的局部修改，可简化为类/函数级别的流程图
- 节点不超过 12 个

```mermaid
{architecture_diagram}
```

---

## 第二章：任务 DAG（或等价拆分）

将实施方案拆成可独立验证的任务序列，使用 Markdown 表格表达依赖与并行关系：

| 任务 | 依赖 | 可并行 | 涉及文件 | 实施步骤 |
|------|------|--------|---------|---------|

要求：
- 每项粒度控制在「估计 2-5 分钟量级」的可交付步骤（允许合并表述，但不得整块模糊）
- 「涉及文件」列出预计改动或触碰的路径（可与真实文件名略有出入，但不得空白）
- 无上游依赖的任务在「可并行」列标注「是」，否则「否」
- 实施步骤编号列出，细化到函数/类/配置层级（与未知细节处标明「待 Step4 终版对齐」类措辞时，须同时在第三章给出对应风险）

---

## 第三章：风险与不确定性分析

至少包含：

1. **关键假设**：列出依赖的技术或组织前提，并标注你认为的证据强度（高/中/低）。
2. **Top 风险**：不少于 **3** 条，可采用表格：`风险描述 | 可能性（高/中/低）| 触发征兆 | 缓解或检测动作`。
3. **与验收的距离**：对照 `P1-requirement.md` 中的验收或成功标准，指出映射薄弱或未覆盖点（若无则明确写「未发现薄弱映射」并给出自查摘要）。

---

## 第四章：回滚策略（草案级）

给出可在实施失败时执行的回滚思路，步骤编号列出；每一步说明作用于哪些改动单元（文件/特性开关/配置）。允许草案级表述，但必须可操作、可理解。

---

## 第五章：方案置信度（草案）

给出总体百分比置信度；若有置信度低于 80% 的决策点，表格列出：决策点、置信度、原因、建议的补强动作。

文末一行：`状态: 提案草案（待 Step2 蒸馏与 Step4 终版收敛）`

---

产出：**仅**写入 `{task_dir}/thinking/P4-proposal-{i}.md`（路径中的 `{task_dir}` 与序号 `{i}` 以 conductor 派发 Task 时的最终替换值为准，须与本提案编号一致）。

质量门禁（自检后才能视为完成）：
- [ ] 接口定义章节存在（或明确「无新增公共接口」）
- [ ] 任务 DAG（或等价结构化拆分表）存在且含依赖与可并行信息
- [ ] 风险与不确定性分析满足本章最低要求（含不少于 3 条风险）
- [ ] 回滚策略章节为非空的可执行草案步骤
- [ ] 全文独立完成：未引用、未猜测其他 `P4-proposal-*.md` 的内容
- [ ] 无 TODO / TBD / 待填写 类占位符
```
