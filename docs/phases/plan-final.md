## 一致性基石与分歧裁决（内化要求，不必单独成章，但必须体现在终版正文）

1. **高置信结论**：从 `P4-consolidated.md` 的趋同点总结（及矩阵中标注为趋同 / 全体趋同的条目）提取结论，作为接口切分、DAG 排序、风险表述的 **默认基石**；在「设计引用」中精炼概括，并点名这些结论源自 consolidated 的高置信合成（可标注 proposal 编号脉络，但不必复述矩阵）。
2. **分歧点**：对 consolidated 中仍为决策焦点或未决项的内容，**须在终版中给出明确取舍**（选 A / 选 B / 分阶段策略 / 显式降级），**禁止**用「视情况而定」「后续再定」等搪塞；若证据不足，须在「方案置信度」中降低对应项置信度并给出补强动作，但 DAG 仍须给出 **当前唯一默认执行路径**。
3. **与 P2 / P1 对齐**：终版不得超出 `P2-design.md` 已确认边界；任务与接口须可映射到 `P1-requirement.md` 的验收或成功标准（薄弱映射须在置信度或预死亡分析中指出）。

## 「对抗审查与采纳说明」章节内容规范

针对 Step3 **全部** `P4-challenge-*.md` 中的关键反驳（含各文件「结构化反驳」中的条目及「## 对抗审查结论」中的要点），编制采纳记录：

- **组织方式**：允许 **逐条** 列出，或 **按主题聚合**（例如「验收映射」「耦合与破坏性变更」「性能安全」「测试回滚」「DAG 粒度」）；聚合时每条记录仍须能拆解对应到具体反驳来源。
- **每条记录必须包含**：
  - **反驳摘要**：一句话或简短列表指回 challenge 中的论点（可用「Challenge m · 反驳条目 n」或复述该条标题）。
  - **结论**：三选一 — **`采纳`** / **`部分采纳`** / **`拒绝`**。
  - **终版体现**：采纳或部分采纳须说明落实位置（例如「见任务 DAG T_k」「见接口定义 xxx」「见预死亡分析第 2 条」）；拒绝须说明 **拒绝理由**（约束、成本、与 P2 冲突、证据不足但接受风险、后置验证计划等）。**部分采纳**须同时说明 **采纳了哪些动作** 与 **未采纳哪些** 及理由。
  - **溯源（强制）**：**至少**标注 **`P4-challenge-{m}.md`** 的编号 **`m`**；若同一结论合并多条反驳，须列出全部相关 **`m`**。
- **覆盖率**：每一个 challenge 文件 **`m`** 至少有一条记录体现对其 **实质性** 反驳的处理结论（可聚合多条反驳为一条记录，但溯源须覆盖该文件）。
- **禁止**：空洞「已阅」、无溯源的结论、或全文仅笼统致谢红队而无逐项/主题级判定。

## 接口定义 / DAG / 预死亡 / 回滚（与 plan.md 质量对齐）

### 接口定义（Gate）

本节必须存在。涉及新增类 / 接口 / 公共 API 时完整定义；不涉及时写「无新增公共接口」。

涉及新增接口时须包含：头文件声明要点（函数签名、参数类型、返回值）、错误码或异常语义、信号槽或异步边界（若适用）、向前 / 向后兼容性说明。

### 任务 DAG

将设计方案拆分为可独立验证的任务序列；表格与模板列一致。

每个任务要求：

- 每步粒度：**估计 2～5 分钟量级**可完成（允许合并表述，禁止整块模糊）
- 「涉及文件」列出预计路径（不得空白）
- 实施步骤细化到 **函数 / 类 / 配置** 层级
- 无依赖任务在「可并行」列标注「是」，否则「否」

### 预死亡分析（Gate）

不少于 **3** 条；每条含可能性（高 / 中 / 低）、检测方式、回滚方案；须与 DAG 中的高风险任务对齐。

### 回滚策略

给出 **可直接执行** 的回滚步骤；任一关键任务失败时可安全回退。
